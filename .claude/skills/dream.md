# dream — 메모리 통합 스킬

사용자의 활동, 결정, 선호도, 컨텍스트를 수집하여 메모리 시스템에 체계적으로 정리합니다.

## 언제 사용할지

- `/dream user` — 최근 세션 활동을 user 메모리에 통합 (역할, 선호도, 지식)
- `/dream feedback` — 이번 대화에서 내린 지시·피드백을 feedback 메모리에 기록
- `/dream project` — 진행 중인 프로젝트 상태, 마일스톤, 결정을 project 메모리에 저장
- `/dream reference` — 외부 도구, 문서, 시스템의 위치를 reference 메모리로 저장

세션이 길어지거나, 중요한 선택이 내려지거나, 패턴이 반복될 때 호출합니다.

## 동작 방식

스킬은 4단계 프로세스를 따릅니다.

### 1단계 — 방향 설정
```
// 메모리 디렉토리 확인
~/.claude/projects/-Volumes-NO-NAME----CAB-STARTER-/memory/
├─ MEMORY.md (인덱스)
├─ user_*.md (역할, 선호도, 지식)
├─ feedback_*.md (지시, 피드백)
├─ project_*.md (진행 상황, 결정)
└─ reference_*.md (도구, 리소스 위치)
```

### 2단계 — 신호 수집
- **이어가기.md** 읽기 (현재 진행 상황 이해)
- **이번 대화 로그** 검토 (새로운 정보, 선택, 피드백)
- **최근 세션 로그** 스캔 (패턴, 반복되는 결정)

### 3단계 — 통합
선택한 메모리 타입에 따라:

**user** 메모리:
- 사용자의 역할, 책임, 지식 수준, 선호도 기록
- 예: "시간 제약이 있음(평일 8-10시간), 탐색보다 출시 우선"
- 포맷: 기본값 시 "Rule / **Why:** / **How to apply:**"

**feedback** 메모리:
- 이번 대화에서 사용자가 준 지시, 정정, 확인 기록
- 예: "스킬은 임의로 생략하지 말고 순서대로 끝까지"
- 포맷: "Rule / **Why:** / **How to apply:**"

**project** 메모리:
- 사업 진행 상황, 마일스톤, 결정, 제약
- 예: "CAB 3주차 대시보드 자동화, 마인드맵 완료"
- 포맷: "Fact / **Why:** / **How to apply:**"

**reference** 메모리:
- 외부 시스템, 도구, 문서 위치
- 예: "선택 기준: business/*.md 에 적힌 것만"
- 포맷: "Resource location and purpose"

### 4단계 — 정리 및 색인화
- **MEMORY.md 업데이트** — 새 메모리의 한 줄 요약 추가
- **상대 날짜 → 절대 날짜** — "지난주" → "2026-06-21"로 변환
- **중복 제거** — 기존 메모리와 내용 비교, 갱신 또는 통합
- **링크 추가** — 관련 메모리는 `[[name]]` 으로 상호 참조

## 체크리스트

스킬 실행 중:

- [ ] **1단계**: 메모리 폴더 구조 확인
- [ ] **2단계**: 이어가기.md + 이번 대화 + 최근 로그 수집
- [ ] **3단계**: 각 메모리 파일에 내용 작성 (frontmatter 포함)
  - `name`: 케밥케이스 슬러그
  - `description`: 한 줄 요약 (미래 대화에서 결정용)
  - `metadata.type`: user / feedback / project / reference
- [ ] **4단계**: MEMORY.md 인덱스 한 줄 추가
- [ ] 상대 날짜 → 절대 날짜 확인
- [ ] 기존 메모리와 중복 체크

## 예시

### /dream user
**입력:** 사용자가 시간 제약과 선호도를 말함

**수집:**
- "평일 8-10시간 대학 행정업무로 묶임"
- "탐색 늘리지 말고 출시로 좁혀라"
- "스킬은 임의로 생략하지 말고 끝까지"

**작성:**
```markdown
---
name: time-constraint-and-shipping-priority
description: 평일 8-10시간 묶여 있어서 탐색 대신 출시 중심 제안 선호
metadata:
  type: user
---

사용자는 평일 대학 행정업무로 **8~10시간이 묶여** 있습니다.

**Why:** 시간이 제한되어 있으므로, 탐색을 늘리는 제안보다 **실행 가능한 구체적 제안** (범위 좁혀서 이번 주에 출시 가능한 것)을 우선합니다.

**How to apply:** 기능을 제안할 때 "뭘 만들지(범위·방향)"는 짧게, "어떻게 만들지(스킬·절차)"는 충실히. 절차 단계를 건너뛰자는 근거로 시간 제약을 쓰지 않기.
```

**색인:**
MEMORY.md에 추가:
```
- [시간 제약과 출시 우선](time-constraint-and-shipping-priority.md) — 평일 묶여있어서 탐색 대신 실행 중심 제안 선호
```

### /dream feedback
**입력:** 사용자가 "스킬 단계를 생략하지 말고 그대로 따르라"고 명시

**작성:**
```markdown
---
name: follow-skills-exactly
description: 스킬·워크플로우는 임의로 줄이지 말고 순서대로 끝까지
metadata:
  type: feedback
---

스킬(skill)·워크플로우가 적용되면, 있는 그대로 · 정해진 순서대로 · 끝까지 따릅니다.

**Why:** 한 번 어겨서 원인 찾는 데 한참 걸렸던 경험. 절차를 임의로 줄이면 도중에 빠뜨린 게 나중에 문제로 떠남.

**How to apply:** "단순해 보인다" "빨리 끝낼 수 있다" "이 정도는 필요 없다"는 생각이 들어도, 스킬이 정한 단계는 끝까지 따릅니다.
```

## 주의사항

- **이어가기.md 우선** — 이 파일이 기준이므로, 대화와 충돌하면 이어가기.md 내용을 믿습니다.
- **상대 날짜 변환 필수** — "지난주", "어제", "다음 목요일" → 절대 날짜로
- **중복 확인** — 새 메모리를 만들기 전에, 기존 파일에 병합할 것은 없는지 확인
- **추측 금지** — 대화·로그에 명시된 것만 기록. 유추하지 않음
- **링크 자유** — `[[name]]`이 기존 메모리가 없어도 괜찮음 (미래 작성 신호)
